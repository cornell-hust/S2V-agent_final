# idea2_v3 数据预处理与训练命令

> 当前 v3 官方链路与 v2 的核心语义保持一致，但文件命名统一为 `compact_trace_v2` 系列。
>
> 官方产物分工：
> - SFT 基础 prepared 文件：`sft_train.compact_trace_v2.jsonl`
> - SFT teacher prepared 文件：`sft_train.teacher_rollout_primary.compact_trace_v2.jsonl`
> - SFT 离线消息缓存：`sft_train.compact_trace_v2.materialized_messages_v1.jsonl`
> - RL train 离线 runtime items：`msad_saver_runtime_train.materialized_items_v1.jsonl`
> - rollout eval / RL eval 离线 runtime items：`msad_saver_runtime_test.materialized_items_v1.jsonl`
> - RL train：`msad_saver_runtime_train.jsonl`
> - rollout eval / RL eval：`msad_saver_runtime_test.jsonl`
> - `*.frame_cache` / `*.feature_cache`：当前 SFT、RL、rollout-eval 的必需输入，不是可选优化

## 1. 官方数据流总览

```text
msad_saver_with_qwen.jsonl
    │
    ├─→ convert_to_saver_agent.py (train)
    │      └─→ msad_saver_runtime_train.jsonl
    │
    ├─→ convert_to_saver_agent.py (test)
    │      └─→ msad_saver_runtime_test.jsonl
    │
    └─→ saver_v3.cli.prepare_sft_manifest
           └─→ sft_train.compact_trace_v2.jsonl
                └─→ sft_train.compact_trace_v2.jsonl.meta.json

optional teacher branch:
    sft_train.compact_trace_v2.jsonl
        └─→ annotate_teacher_judge_sft.py
             └─→ sft_train.teacher_rollout_primary.compact_trace_v2.jsonl
                  └─→ sft_train.teacher_rollout_primary.compact_trace_v2.jsonl.meta.json

runtime train/test
    ├─→ build_frame_cache.py
    └─→ build_feature_cache.py
```

python prepare_materialized_cache.py \

说明：

## 4. 可选：生成 teacher-judge prepared 文件

这一步现在是 v3 官方支持的可选分支，不再是“legacy only”。如果你配置了 `TEACHER_JUDGE_MODEL_PATH`，`scripts/run_full_pipeline.sh` 会自动：
- 先检查并在必要时重建 base prepared：`sft_train.compact_trace_v2.jsonl`
- 优先复用已有的 `sft_train.teacher_rollout_primary.compact_trace_v2.jsonl`
- 如果文件缺失、`.meta.json` 缺失，metadata 与当前 SFT 配置不一致，或 `source_runtime/source_prepared` provenance 过期，则自动重新生成

手动生成命令：

单卡版，适合先做调试或小规模验证：

```bash
python annotate_teacher_judge_sft.py \
  --input data_utils/sft_train.compact_trace_v2.jsonl \
  --output data_utils/sft_train.teacher_rollout_primary.compact_trace_v2.jsonl \
  --model-path /mnt/shared-storage-user/mineru2-shared/zengweijun/Wmh/MLLMs/Qwen3-VL-32B-Instruct \
  --proposal-model-path /mnt/shared-storage-user/mineru2-shared/zengweijun/Wmh/MLLMs/siglip \
  --proposal-torch-dtype auto \
  --include-splits train \
  --input-mode auto \
  --torch-dtype bf16 \
  --device-map auto \
  --attn-implementation flash_attention_3 \
  --max-new-tokens 384 \
  --max-images 8 \
  --topk-frames-per-view 4 \
  --frame-cache-max-cached-videos 64 \
  --batch-size 1
```

本地 8 卡并行版，推荐正式跑全量 teacher-judge 时使用：

```bash
CUDA_VISIBLE_DEVICES=0,1,2,3,4,5,6,7 \
torchrun --standalone --nproc_per_node=8 \
  annotate_teacher_judge_sft.py \
  --input data_utils/sft_train.compact_trace_v2.jsonl \
  --output data_utils/sft_train.teacher_rollout_primary.compact_trace_v2.jsonl \
  --model-path /mnt/shared-storage-user/mineru2-shared/zengweijun/Wmh/MLLMs/Qwen3-VL-32B-Instruct \
  --proposal-model-path /mnt/shared-storage-user/mineru2-shared/zengweijun/Wmh/MLLMs/siglip \
  --proposal-torch-dtype auto \
  --include-splits train \
  --input-mode auto \
  --torch-dtype bf16 \
  --device-map auto \
  --attn-implementation flash_attention_3 \
  --max-new-tokens 384 \
  --max-images 8 \
  --topk-frames-per-view 4 \
  --frame-cache-max-cached-videos 64 \
  --batch-size 1
```

补充说明：
- `torchrun --nproc_per_node=8` 会自动把 teacher-judge 切成 8 个 shard，每个 rank 占用 1 张卡，最后由 rank 0 自动 merge 回最终 `--output` 文件。
- `--batch-size` 表示每张卡内部的 micro-batch，不是全局 batch。上面的 8 卡命令如果写 `--batch-size 2`，含义是每卡 2、总并发约 16。对 32B 多模态 teacher，建议先从 `1` 开始试。

## 7. SFT / RL / Eval 分别用哪些文件

- SFT 基础分支：`data_utils/sft_train.compact_trace_v2.jsonl`
- SFT teacher 分支：`data_utils/sft_train.teacher_rollout_primary.compact_trace_v2.jsonl`
- SFT 实际训练消费：`data_utils/sft_train.compact_trace_v2.materialized_messages_v1.jsonl`
- RL train 原始输入：`data_utils/msad_saver_runtime_train.jsonl`
- RL train 实际消费：`data_utils/msad_saver_runtime_train.materialized_items_v1.jsonl`
- rollout eval 原始输入：`data_utils/msad_saver_runtime_test.jsonl`
- rollout eval / RL eval 实际消费：`data_utils/msad_saver_runtime_test.materialized_items_v1.jsonl`

额外说明：
- `scripts/run_full_pipeline.sh` 会根据是否设置 `TEACHER_JUDGE_MODEL_PATH` 自动决定 SFT 读取 base prepared 还是 teacher prepared。
- 然后在 `Stage 1c` 自动检查并按需重建三份 materialized cache：SFT messages、runtime train items、runtime eval items。
- base prepared、teacher prepared 和全部 materialized cache 都会在进入训练/评测前做严格 metadata 校验；只要配置或 provenance 漂移，pipeline 就会自动重建，而不是静默复用旧文件。

## 9. `use_sample_weights` 是否保留

当前官方 SFT YAML 保持：

```yaml
optimization:
  use_sample_weights: true
```

保留理由：
- VAU 监督样本的难度和信息密度差异很大，既有简单分类，也有多步工具调用与完整事件链监督。
- 不加权时，高频且简单的样本更容易主导损失，削弱事件链完整性和工具轨迹学习。
- 该设置仍然可关闭；如果你要做消融，可以显式改成 `false`。

| 项目 | Fixed Baseline | S2V-Agent / SAVER | 说明 |
|---|---:|---:|---|
| 单次帧数 | 8 | 8 / 次工具调用 | baseline 直接看 8 张 preview；agent 的 scan_timeline/seek_evidence 每次最多 8 帧 |
| 单次总图像数上限 | 8 | 28（上下文同时保留） | baseline 当前配置 max_total_images=8；agent 当前 rollout 配置 max_total_images=28 |
| 轮数上限 | 1 | 14 | baseline 单次直接回答；agent T_max=14 |
| 是否可自适应搜索 | 否 | 是 | baseline 不能追加观察；agent 可继续 scan_timeline/seek_evidence |
| 是否可验证 | 否 | 是 | baseline 无 verify_hypothesis；agent 有显式验证动作 |
| 轨迹累计可接触视觉证据 | ≈ 8 张 | 显著大于 8，理论上最多约 8×若干视觉回合 | agent 虽然任一时刻只保留最多 28 图，但整个轨迹可逐步看更多不同窗口 |
| 是否固定观察 | 是 | 否 | baseline 是 fixed-observation；agent 是 active search |
